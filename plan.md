# SynthAudience — Pre-Flight Campaign Testing with NetLogo + Local LLM Agents

> **Working name:** SynthAudience (alternatives: PreFlight, FocusSim, EchoCrowd — decide later, name doesn't block anything.)
>
> **One-liner:** Test an ad, email, or landing page against a simulated audience of 100–1,000 LLM-powered customer personas living inside a NetLogo social network — see who clicks, who buys, who ignores it and *why* — before spending a single dollar on real traffic.

---

## 1. Problem & Vision

Every new marketing campaign is a paid guess. Small teams spend real budget (ads) or real audience goodwill (email sends) to learn whether creative works. A/B testing tells you *what* won only **after** you paid, and it never tells you *why* the loser lost.

**SynthAudience** flips the order: run the campaign against a synthetic population first.

- Each synthetic customer is a **persona** (demographics + psychographics + buying behavior) whose *reactions* are generated by a small, cheap LLM.
- The personas live as **turtles in a NetLogo model** connected by a social network, so word-of-mouth, social proof, exposure timing, and diffusion emerge from the simulation instead of being guessed.
- The output is a **campaign pre-flight report**: predicted funnel (open → click → convert), segment-by-segment breakdown, verbatim objections ("the subject line sounds like spam", "no price shown, I don't trust it"), and A/B comparison across variants.

This is **Generative Agent-Based Modeling (GABM)**: LLM = agent cognition, ABM = social physics. Neither alone is enough — pure LLM "panels" (just prompting 500 personas) miss network effects and time dynamics; pure rule-based ABMs can't read an actual ad and react to its copy.

### Why this project specifically makes sense for you

1. You already built and own the hardest infrastructure piece: a production-quality **NetLogo MCP server** (JVM bridge, batched state I/O, BehaviorSpace, model authoring). Competitors have to build their sim layer from scratch; you extend yours.
2. It revives your unfinished **ABM course project** with everything you've learned since.
3. It is directly adjacent to your **UNU Macau / Serge Stinckwich collaboration** — the Ebola information/trust model in `docs/research/agent-types-spec.md` is structurally the *same problem* (message exposure → trust-weighted belief update → behavior change → network diffusion). The machinery you build here (LLM-driven per-agent reactions to a "message stimulus") is publishable and reusable there. This project can produce both a **tool** and a **paper**.

**Critical constraints discovered in the codebase (these dictate the architecture):**

1. **One-way bridge.** NetLogo cannot call out to Python/an LLM mid-tick (no callbacks; `py:` extension deliberately blocked in restricted mode). → The LLM loop must be orchestrated **from the Python side**, interleaving `report` → LLM → `command` → `go`.
2. **Single workspace lock.** Every JVM call serializes through one `asyncio.Lock`. Per-agent round-trips would be latency death. → **Batch everything**: ONE `report` pulls all pending agents' state as JSON, ONE `command` writes all decisions back.
3. **The inner loop must live inside the server process**, not go through MCP stdio per agent. The orchestrator is a new module inside `netlogo_mcp`; MCP tools are the *outer* interface (start campaign, step, get report).

## 4. System Architecture

### 4.1 The core design decision: hybrid cognition (this is what makes it feasible)

**Naive design (arXiv 2510.18155, doesn't scale):** every agent calls the LLM every tick. 1,000 agents × 100 ticks = 100,000 LLM calls per run. Dead on arrival.

**Hybrid design (yours):** the LLM is called only at **cognitive events** — the moments a persona actually *encounters* the campaign. Everything between events is classical ABM rules in NetLogo.

- **LLM decides:** first reaction to a stimulus (open? read? click? buy? share? annoyed?), reaction to a friend's recommendation (only at first social exposure), verbatim reasoning.
- **NetLogo decides:** who gets exposed when (channel reach model), how word-of-mouth propagates over the network, attention decay, repeat-exposure fatigue, opinion drift between events (DeGroot-style weighted averaging of neighbor sentiment).

Budget math: 1,000 agents, ~1.3 LLM calls/agent average (not everyone gets socially re-exposed) ≈ **1,300 calls/run**, ~250 output tokens each.
- Local Qwen3-4B on a consumer GPU via Ollama, 8–16 concurrent slots: **10–25 minutes/run**.
- With **archetype caching** (Section 4.4): **1–3 minutes/run**.
- Cloud fallback (Haiku 4.5 / Gemini Flash class): **≈ $0.15–0.50/run**, ~2 minutes.

### 4.3 The simulation loop (precise contract)

Tick = 1 simulated hour (configurable). Per tick, the orchestrator:

1. `command("go")` — NetLogo advances: channel exposure model marks turtles as `exposure-pending` (email delivered, ad impression served, friend shared), applies attention decay, propagates already-decided word-of-mouth.
2. `report(...)` — ONE composed reporter returns JSON list of all `exposure-pending` turtles: `[who, persona-id, exposure-type (direct|social), source-who, prior-sentiment, exposure-count]`.
3. Cognition Engine resolves each event → decision object (Section 5). Async, concurrent, cached.
4. `command(...)` — ONE batched command writes decisions back: `ask turtle 17 [ set state "clicked" set sentiment 0.7 set will-share? true ] ask turtle 42 [...]  ...` (built as a single string; injection-safe because values are typed/escaped by Pydantic first, never raw LLM text).
5. Decisions + verbatims logged to SQLite event store (never into NetLogo — NetLogo holds only numeric/categorical state).

Stop condition: no pending exposures for N ticks, or max ticks reached.

### 4.4 Cost control: archetype caching (the scale unlock)

- Cluster the persona population into **K archetypes** (K ≈ 30–80) by embedding persona cards (demographics+psychographics vector) and k-means.
- **Fidelity modes:**
  - `fast`: LLM called once per (archetype × stimulus × exposure-type); individual agents sample from their archetype's response distribution with persona-conditioned noise. 1,000 agents → ~100–200 LLM calls. Minutes.
  - `full`: LLM called per individual agent (unique verbatims, max diversity). For final runs.
  - `mixed` (default): full fidelity for a stratified sample of ~150 agents (gives verbatims), archetype-cached for the rest.
- Exact-repeat cache: (persona-hash, stimulus-hash, exposure-context-hash) → cached response, persisted in SQLite. Re-running a tweaked campaign only pays for changed exposures. This makes **iterating on copy nearly free** — the core UX promise.

## 5. Cognition Engine (the heart — most design effort goes here)

### 5.1 Persona schema (Pydantic v2)

Modeled on your Ebola `agent-types-spec.md` layered typology (demographics + role + overlays + trust profile):

```python
class Persona(BaseModel):
    id: str
    # Layer 1 — demographics (sampled from real distributions)
    age: int; gender: str; location: str; income_bracket: str
    education: str; occupation: str; household: str
    # Layer 2 — psychographics
    personality: BigFive          # OCEAN, 0-1 floats
    values: list[str]             # e.g. ["frugality", "status", "sustainability"]
    risk_tolerance: float; novelty_seeking: float
    # Layer 3 — consumer profile
    category_involvement: float   # how much they care about this product category
    price_sensitivity: float; brand_loyalty: float
    current_solution: str         # what they use today (status quo = the real enemy)
    pain_points: list[str]; objection_style: str   # skeptic / pragmatist / enthusiast...
    purchase_history_sketch: str  # 2-3 sentence behavioral history
    # Layer 4 — media & social behavior
    channels: dict[str, float]    # email-open propensity, social-scroll speed, ad blindness
    influence: float              # out-influence (maps to network degree)
    susceptibility: float         # in-influence (social proof weight)
    trust_in_ads: float           # baseline skepticism — CRITICAL anti-positivity anchor
    # Layer 5 — narrative card (what the LLM sees)
    persona_card: str             # 120-180 word first-person bio, generated once
```

Population generation: sample layers 1–4 from **explicit joint distributions** (start: hand-designed for the target audience; later: US Census PUMS / Eurostat marginals + iterative proportional fitting for realism). Correlations matter (age↔channel, income↔price-sensitivity) — define a small copula/conditional table, don't sample independently. Persona cards generated once by the LLM from the structured fields, then frozen and versioned. Optionally seed diversity from **PersonaHub** (Tencent's open persona corpus) filtered to the target segment.

Audience spec is a YAML file the user writes (or the MCP client generates conversationally):

```yaml
audience: "bootstrapped SaaS founders"
size: 500
distributions:
  age: {25-34: 0.45, 35-44: 0.35, 45-54: 0.20}
  income_bracket: {mid: 0.5, high: 0.4, low: 0.1}
  trust_in_ads: {mean: 0.25, sd: 0.15}   # founders are ad-cynical
network: {topology: watts-strogatz, k: 8, rewire: 0.1, degree_by: influence}
```

### 5.2 Reaction prompt (two-stage, forced-choice, anti-sycophancy)

**Stage 1 — Attention gate** (cheap, short): persona card + only what's visible pre-click (subject line + sender for email; thumbnail + headline for an ad) + context ("you're clearing your inbox on a Tuesday morning, 40 unread"). Output: `open/scroll_past/delete`. This stage kills the LLM positivity bias structurally — most real reactions die here, so make the model choose *before* it reads the persuasive body.

**Stage 2 — Full reaction** (only for openers): full stimulus content. Output schema:

```python
class Reaction(BaseModel):
    attention_seconds: int                  # 1-120, how long they'd actually look
    action: Literal["ignore","click","save_for_later","buy","share","unsubscribe","report_spam"]
    sentiment: float                        # -1..1
    trust_delta: float                      # brand trust change
    reason: str                             # verbatim, first person, <40 words
    objections: list[str]                   # empty if none
    would_share_with: Literal["nobody","close_friends","publicly"]
```

Enforced via **structured outputs** (Ollama `format: json_schema` / OpenAI-compatible `response_format`), validated by Pydantic, retried once on failure.

**Bias mitigations (each is a known failure mode of naive persona prompting — this list is your 61%→86% ladder):**

1. **Base-rate anchoring in the prompt:** "Most emails like this get deleted in under 2 seconds. React as this specific person actually would, not politely."
2. **Reasons-before-action ordering** in the schema (forces grounding before the choice).
3. **`ignore` is the default framing**, action requires justification — never "would you like to click?"
4. **Persona-conditioned skepticism:** `trust_in_ads` and `objection_style` injected verbatim into the system prompt.
5. **Temperature 0.8–1.0 + persona-seeded sampling** for distributional spread (temperature 0 collapses everyone to the modal answer).
6. **Calibration layer (Section 8):** raw LLM action propensities pass through a fitted logistic mapping to real-world base rates. The LLM provides *relative* discrimination (which persona, which variant); calibration provides *absolute* levels.
7. **No leading context:** the stimulus is presented as-is, never described as "our great new campaign".

### 5.3 Social re-exposure

When a neighbor shares, the exposed agent gets ONE more LLM call with modified context: "Your friend [archetype-of-source], whose taste you [trust-level], shared this with the comment: [source's reason verbatim]." Social proof effect emerges from the model, not from a hard-coded multiplier. Subsequent same-source exposures are rule-based (diminishing returns in NetLogo).

## 6. NetLogo Model (`market_sim.nlogox`)

Generated programmatically via your existing `create_model` tool. Keep it deliberately simple — NetLogo does physics, not psychology.

- **Turtles** = personas. Variables: `persona-id, state, sentiment, trust, exposure-count, exposure-pending?, exposure-type, source-who, decided-tick, will-share?, attention-decay`.
- **States (funnel):** `unaware → exposed → engaged (opened/viewed) → clicked → converted | ignored | annoyed (unsub/spam)`. Color-coded for the GUI demo (gray → yellow → orange → green / red).
- **Links** = social ties. Topology options: Watts–Strogatz small-world (default, communities), Barabási–Albert (influencer-heavy audiences), or loaded from a real follower-graph sample. Degree assigned by persona `influence`.
- **Channel exposure procedures (per tick):**
  - `email`: all list members receive at send-tick(s); open-gate resolved by cognition stage 1 with persona `channels.email` propensity as prior.
  - `paid-social`: impressions/tick = budget-derived reach curve; targeting = filter on persona attributes (mirrors real ad targeting); frequency capping.
  - `organic/WOM`: sharers trigger neighbor exposure with delay 1–6 ticks.
- **Between-event dynamics (pure rules):** sentiment drift toward neighbor mean (DeGroot weight = `susceptibility`), attention decay, fatigue penalty on repeat exposures (each repeat lowers stage-1 open prior).
- **Plots/monitors** (auto-created via your widget support): funnel counts, cumulative conversions, WOM share of exposures, sentiment histogram.
- **BehaviorSpace experiment** baked into the model: N replicate seeds × variant → your existing `run_experiment` gives **confidence intervals for free** (headless, threaded, separate JVM — doesn't block the GUI workspace).

## 7. Stimulus Ingestion

Everything the audience "sees" is normalized to one schema:

```python
class Stimulus(BaseModel):
    id: str; variant_of: str | None
    type: Literal["email","social_ad","landing_page","search_ad"]
    # pre-click surface (stage-1 material)
    teaser: dict          # email: subject+sender+preheader; ad: headline+thumbnail-description
    # full content (stage-2 material)
    body_text: str        # extracted/cleaned copy
    cta: str; offer: str | None; price_shown: str | None
    visual_description: str | None   # from a VLM pass, or hand-written
    channel_params: dict  # send_tick, budget, targeting filter
```

- **v1 (MVP):** user pastes copy / fills YAML. Zero parsing risk.
- **v2:** email `.html` → text via `beautifulsoup4` + `html2text`; landing page URL → fetch + readability extraction.
- **v3:** image ads → description via a small vision model (Gemma-3-4B is multimodal and already in your Ollama stack; or Claude Haiku for quality). The personas read the *description* — small text LLMs stay the cognition engine.

A/B testing is native: a campaign = list of stimuli sharing `variant_of`; the orchestrator runs each variant against **the identical frozen population and identical random seeds** (paired design → much tighter variant comparisons than independent runs).

## 8. Analytics, Calibration & the Report

### Event store
SQLite (stdlib, zero deps) with tables: `runs`, `personas`, `exposures`, `decisions` (incl. verbatims), `tick_metrics`. Parquet export for pandas analysis.

### The Pre-Flight Report (the actual product)
Generated as markdown + HTML per campaign:

1. **Funnel** per variant with 95% CIs from replicate runs (reach → attention → engagement → click → convert → share / unsub).
2. **Segment heatmap**: conversion & sentiment by archetype/age/price-sensitivity — *who* the campaign lands with and who it repels.
3. **Objection mining**: cluster verbatim `reason`/`objections` by embeddings (`nomic-embed-text` via Ollama, or sentence-transformers MiniLM) → top 5 objection themes with representative quotes and affected-segment counts. **This is the "why" that real A/B tests never give you.**
4. **Weak-point diagnosis**: where the funnel leaks most vs. archetype expectations (e.g. "82% die at the subject line for skeptic archetypes").
5. **Variant verdict**: paired comparison, effect size, "variant B wins on click but loses on trust with high-income segment".
6. **WOM forecast**: share rate, network amplification factor, whether organic spread is plausible.

### Calibration layer (what separates a toy from a tool)
- Maintain `calibration.json`: per channel, a logistic mapping from raw LLM propensity → calibrated probability, fitted on whatever real data exists (industry benchmark base rates as the floor: e.g. cold-email open ~20–35%, CTR ~2–5%, display CTR ~0.1–0.5%; user's own past campaign stats when available — a tiny `calibrate --from-csv` command ingests a past campaign's stats and refits).
- Report both **absolute (calibrated)** and **relative (uncalibrated)** numbers, clearly labeled. Relative variant ranking is trustworthy earlier than absolute prediction — say so in the report footer, always.

## 10. Validation Plan (make-or-break for credibility AND the paper)

Three rungs, in order. Every claim in the README traces to one of these.

**Rung 1 — Face validity / classical replication (Phase 3)**
- Reproduce the **Bass diffusion curve** shape from WOM-only spread. If your synthetic audience can't produce an S-curve, nothing else matters.
- Reproduce known monotonic effects: price ↑ → conversion ↓; social proof → lift; frequency ↑ → fatigue. Sensitivity sweeps via `run_experiment`.
- **Distributional sanity:** response distributions must NOT collapse to the mode (measure entropy across personas; compare against temperature/seed).

**Rung 2 — Backtest vs public datasets (Phase 5)**
- **Hillstrom MineThatData Email Challenge** (64k customers, real e-commerce email campaign, treatment/control, opens/clicks/conversions, with customer covariates) — the single best public target: build personas from the covariate distribution, simulate the send, compare predicted vs real open/click/conversion **by segment**. This is your headline number.
- Kaggle marketing A/B datasets (ad vs PSA) — test whether the sim ranks the variants correctly.
- Success metric: **directional agreement on segment ranking + variant ranking** first; absolute rates within calibrated bounds second. (Match Artificial Societies' framing: distribution accuracy %, response consistency %.)

**Rung 3 — Robustness audit (per arXiv 2605.18890, Phase 5)**
- Sensitivity to: model choice (Qwen vs Phi vs Haiku), temperature, prompt paraphrase, persona-card wording, seed. Report variance decomposition.
- Ablations: no-network vs network, no-calibration vs calibrated, archetype-cache vs full fidelity (quantify what caching costs in accuracy).
- Pre-register the backtest predictions (commit predictions before scoring) — cheap, and it converts "demo" into "science".

**Rung 4 (stretch) — one real pilot:** find one small business / creator newsletter willing to share a past campaign's stats; blind-predict, then compare. One real case study beats all benchmarks for the README.

## 13. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| **LLM homogeneity** — all personas react the same (the classic GABM failure) | High if naive | Phase 0 tests this FIRST; forced-choice + base-rate anchoring + temperature + persona-conditioned skepticism; entropy metric in CI |
| **Positivity bias** — synthetic audience loves everything | High if naive | Stage-1 attention gate kills most reactions pre-persuasion; `ignore` default framing; calibration floor to industry base rates |
| **Absolute numbers are wrong** → users misled | Medium | Always dual-report relative vs calibrated-absolute; report footer disclaimer; calibration from user's own data |
| **Throughput too slow locally** | Medium | Archetype caching (10× fewer calls); fidelity modes; concurrency; cloud fallback is cents per run |
| **JVM lock serialization bottleneck** | Low (design handles it) | Batched read/write contract — 2 JVM calls per tick regardless of agent count |
| **NetLogo command injection from LLM output** | Low | LLM output never becomes NetLogo code — Pydantic-typed values only, formatted by trusted templates; keep `NETLOGO_MCP_RESTRICTED` |
| **Scope creep** (this plan is big) | High | Phases are strictly ordered; each ends runnable; Phase 0 gate before infra; resist the dashboard until Phase 4 |
| **"Someone already did it"** demotivation | — | Nobody has shipped the open-source, local-first, network-dynamics version. The 61%→86% gap is machinery, and the machinery is exactly this plan |
| **Validation shows weak accuracy** | Possible | Still publishable (honest negative/mixed results in GABM are valued — cite the robustness-audit paper); relative variant ranking is a lower bar likely to hold |
